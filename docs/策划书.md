# XiMonitor 策划书

## 1. 项目目标

XiMonitor 是一个使用 Rust 开发的轻量级服务器监控面板，整体形态参考哪吒面板：

- 服务端作为中心节点部署，可通过网页访问
- 被监控端通过一条简单命令或一个 agent 二进制快速接入
- 网页端默认只读，不在网页中提供写配置能力
- 所有可编辑信息统一放在系统配置文件中
- 优先保证稳定性、可维护性和防御性编程

首版重点放在“服务端本体 + 可接入 agent + 只读监控页面”，不追求复杂告警、计费、工单等外围能力。

## 2. 核心需求

首版必须支持以下能力：

- 展示所有已连接子机的在线状态
- 展示 CPU 占用率
- 展示最近负载
- 展示内存占用
- 展示磁盘占用，支持多磁盘/多挂载点
- 展示网络总上传、总下载
- 展示当前上传速度、当前下载速度
- 展示服务端到子机的连接时延
- 支持节点掉线识别
- 支持通过配置文件控制监听地址、鉴权令牌、刷新周期等参数

## 3. 非目标

以下内容不放在首版：

- 网页端登录后台与复杂 RBAC
- 浏览器中直接编辑节点配置
- 节点分组拖拽、可视化大屏主题编辑
- TCP/UDP 端口探针、站点拨测、日志采集
- 集群级别高可用
- 多租户

这些能力保留为后续迭代项。

## 4. 系统架构

项目采用 Rust workspace，拆分为 3 个 crate：

1. `ximonitor-proto`
   负责共享数据结构、协议消息、配置模型中的公共部分。

2. `ximonitor-server`
   负责配置加载、WebSocket 接入、节点状态维护、HTTP API、只读网页渲染。

3. `ximonitor-agent`
   负责采集被监控机指标、维持与服务端连接、回报实时状态、统计网络速率。

整体拓扑如下：

```text
agent(node-a) ----\
agent(node-b) ----- WebSocket ---- ximonitor-server ---- HTTP ---- browser(read-only)
agent(node-c) ----/
```

## 5. 通信模型

首版建议使用单条长连接 WebSocket 作为 agent 和服务端之间的主通信通道。

原因：

- 服务端可以准确知道“在线/离线”
- 可以使用 ping/pong 或应用层心跳测量实时延迟
- 可以减少 agent 反复建连带来的额外开销
- 协议后续扩展更方便

### 5.1 鉴权方式

首版采用配置文件中的共享令牌鉴权：

- agent 启动时携带 `node_id`、`token`
- 服务端校验 token
- 校验通过后允许建立连接

后续可扩展：

- 每节点单独 token
- 首次注册签发凭据
- mTLS

### 5.2 协议消息

协议建议使用 JSON，优先降低调试成本。后续如果性能成为瓶颈，再考虑二进制编码。

主要消息类型：

- `hello`
  agent 首次上线时发送节点基本信息

- `metrics`
  agent 周期性发送监控指标

- `ping`
  服务端主动发送延迟测量包

- `pong`
  agent 回应延迟测量包

- `server_notice`
  服务端向 agent 下发只读提示或调试信息

## 6. 指标设计

### 6.1 节点基础信息

- 节点 ID
- 主机名
- 操作系统
- 内核版本
- CPU 型号
- CPU 核心数
- agent 版本
- 启动时间

### 6.2 实时监控指标

- CPU 总占用率
- 1/5/15 分钟负载
- 总内存、已用内存、可用内存
- 总交换分区、已用交换分区
- 系统运行时长
- 多磁盘/多挂载点空间信息
- 网络总接收/总发送字节数
- 网络当前接收/发送速率
- 最近上报时间
- 服务端测得往返延迟

### 6.3 多磁盘展示模型

每个挂载点单独一条记录：

- 设备名
- 挂载点
- 文件系统类型
- 总容量
- 可用容量
- 已用容量
- 已用百分比

默认过滤临时文件系统，例如：

- `tmpfs`
- `devtmpfs`
- `overlay`

过滤规则放进配置文件，便于调整。

## 7. 服务端设计

### 7.1 职责

- 加载并校验配置文件
- 接收 agent 连接
- 管理节点在线状态
- 维护最新指标快照
- 提供只读 JSON API
- 提供只读 Web 页面
- 对长时间未上报的节点判定为离线

### 7.2 内部模块

- `config`
  配置文件解析、默认值、校验逻辑

- `protocol`
  协议消息编解码

- `registry`
  节点连接注册表

- `state`
  节点状态快照与聚合视图

- `web`
  HTTP 路由、页面、静态资源

- `latency`
  ping/pong 延迟测量

### 7.3 状态存储

首版以内存存储为主，配合可选 JSON 快照落盘：

- 内存中保存最新状态，保证页面访问速度
- 周期性将快照写入本地文件，便于进程重启后恢复最近状态
- 如果快照损坏，服务端应跳过损坏文件并重新初始化，不得 panic

后续如果需要历史曲线，可引入 SQLite / Parquet / TSDB。

### 7.4 只读网页

网页端仅负责展示，不负责修改：

- 总览页：在线节点数、离线节点数、总流量、平均延迟
- 节点列表页：每台机器的关键指标卡片
- 节点详情页：磁盘、内存、负载、网络明细

前端建议采用：

- 服务端内嵌静态 HTML/CSS/JS
- 页面通过 `/api/*` 拉取 JSON
- 自动轮询刷新

这样构建简单，部署成本低，也更符合“服务端本体优先”的目标。

## 8. Agent 设计

### 8.1 职责

- 读取本地配置
- 采集系统指标
- 计算网络速率
- 与服务端保持连接
- 自动重连
- 响应服务端 ping

### 8.2 指标采集策略

Linux 首版优先直接读取系统接口：

- `/proc/stat`
- `/proc/loadavg`
- `/proc/meminfo`
- `/proc/uptime`
- `/proc/net/dev`
- `/proc/mounts`

磁盘容量通过系统调用获取，不依赖 shell 命令。

这样做的优点：

- 部署简单
- 外部依赖少
- 更可控，便于防御性处理异常输入

### 8.3 部署方式

首版提供两种方式：

1. 直接运行
   适合容器、调试或手动部署

2. 安装为 systemd 服务
   提供一个简化安装命令或 agent 自带安装子命令

目标体验示例：

```bash
ximonitor-agent install \
  --server ws://monitor.example.com:8080/ws \
  --node-id hk-01 \
  --token YOUR_TOKEN
```

安装命令负责：

- 写入 agent 配置文件
- 生成 systemd unit
- 启用并启动服务

网页端不提供安装入口，所有安装参数均由运维在配置文件或命令行中管理。

## 9. 配置文件设计

### 9.1 服务端配置

建议使用 TOML。

示例：

```toml
[server]
listen = "0.0.0.0:8080"
public_base_url = "http://127.0.0.1:8080"
node_registry_path = "./config/server.json"
snapshot_path = "./data/snapshot.json"
stale_after_secs = 20
ping_interval_secs = 10

[ui]
page_title = "XiMonitor"
refresh_interval_secs = 5

[filters]
ignored_filesystems = ["tmpfs", "devtmpfs", "overlay"]
```

### 9.2 Agent 配置

```toml
[agent]
node_id = "hk-01"
node_label = "Hong Kong 01"
server = "ws://monitor.example.com:8080/ws"
token = "change-me"
report_interval_secs = 5
```

## 10. 防御性编程要求

这是首版必须贯彻的工程原则。

### 10.1 输入校验

- 所有配置字段必须校验范围和值类型
- 空字符串、超长字符串、非法地址都要给出明确错误
- agent 上报的数值必须做边界检查

### 10.2 网络健壮性

- 连接断开自动指数退避重连
- WebSocket 消息大小限制
- 消息解析失败只影响当前连接，不拖垮服务进程
- 心跳超时后要清理连接状态

### 10.3 文件健壮性

- 配置文件缺失时给出明确错误
- 快照文件损坏时允许服务继续启动
- 落盘采用“先写临时文件再原子替换”

### 10.4 并发安全

- 所有共享状态通过受控结构访问
- 避免无界 channel、无界缓存
- 所有后台任务要可取消、可收敛

### 10.5 可观测性

- 统一日志格式
- 明确区分 `info`、`warn`、`error`
- 关键错误带上下文信息

## 11. 安全边界

首版安全策略：

- agent 只允许访问受保护的 WebSocket 接口
- Web 页面只读，不包含任何写操作 API
- 所有管理类参数通过本地配置文件维护
- 建议实际部署时放在 Nginx / Caddy 之后，并开启 HTTPS

后续可以增加：

- IP 白名单
- 单节点令牌
- mTLS
- 后台登录保护

## 13. 当前建议的落地顺序

为了尽快看到结果，建议我们按下面顺序推进：

1. 先完成可运行的服务端和协议
2. 再完成 Linux agent 指标采集
3. 然后做只读页面和节点详情展示
4. 最后补安装脚本、快照持久化和文档

这样能尽快把“能连上、能看到、能判断在线状态”的主链路跑通。

## 14. 风险与应对

### 风险 1：不同 Linux 发行版的磁盘/网卡口径不同

应对：

- 首版以常见 Linux 为主
- 做好文件缺失和字段异常处理
- 通过配置支持过滤特殊文件系统

### 风险 2：网络速率口径抖动

应对：

- 以两次采样间隔差值计算速率
- 对异常回退值直接丢弃本次速率

### 风险 3：页面数据量增大后前端刷新压力上升

应对：

- 首版先使用轮询
- 后续可替换为 SSE / WebSocket 推送
